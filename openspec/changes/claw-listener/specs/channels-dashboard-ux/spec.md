## ADDED Requirements

### Requirement: Channels tab in settings page
The TrustedClaw settings page SHALL have a new "Channels" tab that serves as the control center for managing WhatsApp and Telegram connections. The tab SHALL show real-time connection status for each channel.

#### Scenario: Channels tab visible
- **WHEN** user navigates to Settings
- **THEN** a "Channels" tab appears alongside "Context Prompt", "Skills", "Gates", and "API Keys"
- **THEN** the tab shows a card for each connected or available channel

### Requirement: WhatsApp connection flow with QR code in dashboard
The Channels tab SHALL allow connecting WhatsApp by displaying a QR code directly in the dashboard UI. The QR code SHALL be generated by the EC2 ClawListener and delivered via the Supabase Realtime command channel.

#### Scenario: Initial WhatsApp connect — QR code display
- **WHEN** user clicks "Connect WhatsApp" and no active WhatsApp session exists
- **THEN** dashboard sends `request_qr` command via `listener_commands` table
- **THEN** EC2 responds with a QR code string
- **THEN** dashboard renders the QR code using `qrcode.react` library in a prominent card
- **THEN** instructions are shown: "Open WhatsApp > Settings > Linked Devices > Scan this code"

#### Scenario: QR code auto-refresh
- **WHEN** the QR code expires (WhatsApp regenerates every ~60 seconds)
- **THEN** EC2 automatically sends a new QR string via `listener_responses`
- **THEN** dashboard updates the displayed QR code without user action
- **THEN** a countdown timer shows time until next refresh

#### Scenario: Successful WhatsApp connection
- **WHEN** user scans the QR code successfully
- **THEN** EC2 sends `{ status: "connected", phone: "+972..." }` via gate metadata update
- **THEN** dashboard immediately shows green "Connected" badge with the phone number
- **THEN** the QR code view is replaced with the connected status card

### Requirement: WhatsApp connected state display
When WhatsApp is connected, the channel card SHALL show: connection status badge (green "online"), phone number, last message timestamp, total message count, listener status, and action buttons.

#### Scenario: Connected WhatsApp card
- **WHEN** WhatsApp is connected and the Channels tab is open
- **THEN** the card shows: green "Connected" badge, phone number (e.g., "+972-50-1234567"), "Last message: 2 min ago", "Messages: 1,234", "Listener: online (EC2 eu-west-1)"
- **THEN** action buttons shown: "Disconnect", "Rescan History"

### Requirement: Telegram connection flow with bot token input
The Channels tab SHALL allow connecting Telegram by entering a bot token. The token is validated by the EC2 ClawListener.

#### Scenario: Telegram bot token setup
- **WHEN** user opens the Channels tab and no Telegram bot is connected
- **THEN** a card shows a password input field for the bot token
- **THEN** helper text reads: "Get a token from @BotFather on Telegram"
- **THEN** a "Connect Bot" button submits the token

#### Scenario: Successful Telegram connection
- **WHEN** user enters a valid bot token and clicks "Connect Bot"
- **THEN** dashboard sends `connect_telegram` command with the token via `listener_commands`
- **THEN** EC2 validates the token, starts the bot, responds with `{ status: "connected", bot_username: "@MyBot" }`
- **THEN** dashboard shows green "Connected as @MyBot" with message stats

#### Scenario: Invalid Telegram token
- **WHEN** user enters an invalid or revoked token
- **THEN** EC2 responds with `{ status: "error", message: "Invalid bot token" }`
- **THEN** dashboard shows a red error message below the input

### Requirement: Telegram connected state display
When Telegram is connected, the channel card SHALL show: connection status, bot username, last message timestamp, message count, and action buttons.

#### Scenario: Connected Telegram card
- **WHEN** Telegram bot is connected
- **THEN** the card shows: green "Connected" badge, bot username (e.g., "@MyClawBot"), message stats, listener status
- **THEN** action buttons: "Disconnect", "Test Connection"

### Requirement: Disconnect channels from dashboard
The user SHALL be able to disconnect any channel from the Channels tab. Disconnection SHALL gracefully stop the bridge on EC2.

#### Scenario: Disconnect WhatsApp
- **WHEN** user clicks "Disconnect" on a connected WhatsApp channel
- **THEN** dashboard sends `disconnect_whatsapp` command
- **THEN** EC2 stops the WhatsApp bridge and logs out the session
- **THEN** dashboard shows the channel as "Disconnected" with a "Connect" button

#### Scenario: Disconnect Telegram
- **WHEN** user clicks "Disconnect" on a connected Telegram channel
- **THEN** dashboard sends `disconnect_telegram` command
- **THEN** EC2 stops the Telegram bot polling
- **THEN** dashboard shows the channel as "Disconnected" with token input

### Requirement: Real-time status updates via Supabase Realtime
The Channels tab SHALL subscribe to Supabase Realtime on the `gates` table (filtered by user_id) to receive live status updates from EC2 heartbeats.

#### Scenario: Live heartbeat display
- **WHEN** EC2 sends a heartbeat (every 5 minutes) updating gate metadata
- **THEN** dashboard immediately reflects the updated status (online/reconnecting/disconnected), message count, and last heartbeat timestamp
- **THEN** no manual refresh is needed

#### Scenario: EC2 goes offline
- **WHEN** EC2 stops sending heartbeats for more than 10 minutes
- **THEN** dashboard shows yellow "Last seen: 10 min ago" warning
- **THEN** after 30 minutes without heartbeat, status changes to red "Offline"

### Requirement: Rescan history from dashboard
The user SHALL be able to trigger a historical message rescan from the Channels tab.

#### Scenario: Rescan WhatsApp history
- **WHEN** user clicks "Rescan History" on a connected WhatsApp channel
- **THEN** dashboard sends `rescan_history` command with `{ hours: 24 }`
- **THEN** EC2 fetches messages from the last 24 hours and ingests them
- **THEN** dashboard shows progress: "Rescanning... Found 234 messages, ingested 198 new"

### Requirement: Add new channel button
The Channels tab SHALL have an "Add Channel" button that presents available channel types (WhatsApp, Telegram) and initiates the connection flow.

#### Scenario: Add new channel
- **WHEN** user clicks "Add Channel"
- **THEN** a selection appears with channel type options (WhatsApp, Telegram)
- **THEN** selecting a type initiates the appropriate connection flow (QR for WhatsApp, token for Telegram)
- **THEN** a new gate is created in Supabase for the selected type
